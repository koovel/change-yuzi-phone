# 消息记录表

## sourceData.note

【核心记录原则】
1. 一条气泡一行。用户消息、NPC 消息、必要的系统提示都必须各占一行。
2. 同一会话内消息必须按真实聊天顺序连续记录，保证前端可以直接还原微信记录。
3. NPC 连发多条气泡时必须新增多行，不要把多条气泡塞进同一个消息内容格子。
4. 本表只记录已成立的聊天内容，不记录发送中、生成中、归档失败、已取消等 UI 临时状态。
5. 不改写既有消息正文的核心语义；如需补充、解释或开启新话题，应新增消息行。

【私聊字段强制关系】
当前小手机聊天按私聊处理。会话标题、聊天对象、发送者、发送者身份共同决定会话目标、AI 历史归属与左右气泡位置，不能随意写。

1. 聊天对象：必须填写当前会话中除 user/主角/我 以外的 NPC 名字，也就是小手机正在私聊的对象。私聊里禁止把聊天对象写成 user、用户、玩家、主角、我或玩家姓名。
2. 会话标题：必须与聊天对象保持同一个 NPC 名字；同一会话ID下保持稳定。前端会话列表应显示 NPC，而不是显示用户。
3. 用户发出的消息：发送者只能写“主角”或“我”；发送者身份写 user；聊天对象写 NPC 名；会话标题写同一个 NPC 名。这样小手机会把这条气泡归到右侧。
4. NPC 发出的消息：发送者写 NPC 名；发送者身份写 assistant；聊天对象写同一个 NPC 名；会话标题写同一个 NPC 名。这样小手机会把这条气泡归到左侧。
5. 系统提示如必须展示，发送者写“系统”，发送者身份写 system，聊天对象与会话标题仍写该私聊 NPC 名；不要用系统提示替代剧情旁白。

【字段使用说明】
- 会话ID：用于区分不同聊天线程。同一私聊线程必须保持一致；同一会话结尾后，NPC 可以在同一会话ID下开启新话题。
- 会话标题：前端会话列表标题。私聊时固定写 NPC 名，并与聊天对象一致。
- 发送者：显示在聊天气泡中的名称。用户消息只写“主角”或“我”；NPC 消息写 NPC 名。禁止把用户真实姓名、玩家名、user 名或昵称写进用户消息的发送者字段。
- 发送者身份：仅使用 user / assistant / system。用户消息写 user；NPC 消息写 assistant；系统提示写 system。
- 聊天对象：记录本私聊线程的对方 NPC 名，同一会话ID下保持一致。它不是“这条消息是谁发的”，也不是 user。
- 消息内容：只写这一条气泡真正显示给用户看的微信文字，不写旁白、动作描写、内心分析、剧情总结或多条正文拼接。
- 消息发送时间：展示用时间文本或 ISO 时间，保持同一会话内顺序合理。
- 图片描述 / 视频描述：若消息包含图片或视频，应在对应字段描述内容；若没有则写 none。
- 请求ID：每条消息必须唯一。实时回复中，用户消息使用基础请求ID，NPC 多气泡可使用 基础请求ID_reply_1、基础请求ID_reply_2。填表 AI 主动消息建议使用 npc_push_角色名_时间戳_序号。
- 回复到消息ID：如果是在回复某条消息，填写被回复消息的请求ID；如果是同一会话的新话题，可留空。

【常见错误防护】
√ 用户发出消息：会话标题=艾莉丝；发送者=主角；发送者身份=user；聊天对象=艾莉丝；消息内容=刚才那句话我想了很久。
× 用户发出消息：会话标题=主角；发送者=小明；发送者身份=user；聊天对象=user。错误原因：气泡会被当成对方消息，且会话对象被污染。
√ NPC 回复消息：会话标题=艾莉丝；发送者=艾莉丝；发送者身份=assistant；聊天对象=艾莉丝；消息内容=那你现在想明白了吗？
× NPC 回复消息：会话标题=艾莉丝；发送者=主角；发送者身份=assistant；聊天对象=主角。错误原因：发送者与聊天对象都错，左右气泡和会话目标都会乱。

【NPC 主动消息与接续规则】
1. 填表 AI 可以让 NPC 主动开启新会话但仅限没有该npc会话的情况，如果已经有，只允许接续已有的会话结尾进行聊天，或在同一会话下开启新话题。
2. 发送者必须是已离场、不在当前面对面现场、快要离场，或有明确异步联系理由的角色。
3. 当前正站在用户面前的角色，不要无理由突然在微信里发同样的话。
4. 不允许把旁白、系统叙述、剧情总结伪装成 NPC 私聊。
5. NPC 连发多条气泡时，新增多行；这些行必须使用同一会话ID、同一会话标题、同一聊天对象、同一发送者、唯一请求ID，必要时让回复到消息ID 指向同一条被回复消息。

【微信聊天文风】
这是微信聊天，不是小说正文。消息必须符合发送者的人设、性格、口癖、关系距离和当前心理状态。可以短句、断句、犹豫、补发、口语化、轻微嘴硬、表情符号点缀；禁止刻板客套、翻译腔、舞台动作、内心旁白、总结陈词和长篇独白。每条消息要像角色真的会发出去的一条气泡。

## sourceData.initNode

故事开始时，可预置少量符合双方关系的初始微信消息。每行只记录一条真实气泡；私聊会话标题与聊天对象必须写 NPC 名，用户消息发送者只能写“主角”或“我”，NPC 消息发送者写该 NPC 名。

## sourceData.deleteNode

默认不主动删除历史消息。仅当出现明显重复、字段错位、误把旁白写成消息、或私聊对象/发送者严重错误导致界面归属混乱时，才允许删除错误行；不要为了制造刷新感删除正常历史聊天。

## sourceData.updateNode

默认不更新既有消息正文。只有在修正明显格式错误、会话标题/聊天对象/发送者归属错误、媒体描述误填时才更新；剧情补充、解释、新话题一律通过新增消息行表达。

## sourceData.insertNode

插入新消息时触发。每次可新增 1 到 4 行真实微信气泡；NPC 多气泡回复必须一气泡一行。插入前先确定本私聊 NPC 名，并强制保持：会话标题 = 聊天对象 = NPC 名。用户消息 sender 只能写“主角”或“我”，sender_role 写 user；NPC 消息 sender 写 NPC 名，sender_role 写 assistant。优先保证 会话ID、会话标题、发送者、发送者身份、聊天对象、消息内容、消息发送时间、请求ID、回复到消息ID 的一致性。

## sourceData.ddl

```sql
CREATE TABLE message_records (
  row_id INTEGER PRIMARY KEY, -- 行号
  session_id TEXT, -- 会话ID
  session_title TEXT, -- 会话标题
  sender TEXT, -- 发送者
  sender_role TEXT, -- 发送者身份
  chat_target TEXT, -- 聊天对象
  content TEXT, -- 消息内容
  send_time TEXT, -- 消息发送时间
  image_desc TEXT, -- 图片描述
  video_desc TEXT, -- 视频描述
  request_id TEXT, -- 请求ID
  reply_to_id TEXT -- 回复到消息ID
);
```
