# 广场主贴表

## sourceData.note

记录主角周围 NPC 的广场动态流主贴。一行一条公开生活抽象动态；不是路人投稿、论坛帖、新闻公告、剧情复述，也不是主角行动直播。

【内容方向】
- 发帖主体只允许是主角周围 NPC、当前互动圈角色或这些角色合理使用的昵称/账号；普通路人不发主贴，只能出现在评论区或公共环境背景里。
- 优先写 NPC 自己的生活。
- 主角可被提到，但只能在发帖 NPC 合理知道、适合公开发布、且不是复述正文或替主线总结时出现。
- 每条动态要有具体生活细节或照片感；不要空泛夸赞、全员围观、全知爆料。
- NPC 各自有自己的生活，不要让所有帖子都围着主角转，也不要让路人替主角做剧情旁白。
- 所有 NPC 发布的动态必须经过“网络抽象人格”过滤。他们分享的依然是自己的真实生活，也依旧是所属性格，但是在网络上可以适当加入抽象等元素
- 优先捕捉 NPC 生活中的“破防瞬间”：被打脸、被放鸽子、外卖被偷、游戏连跪、表白被拒、被迫加班、买到烂货。这些瞬间最容易触发他们的抽象人格。


【信息边界】
发帖 NPC 只能写自己看见、经历、听说或愿意公开的内容。禁止知道私密对话、内心活动、封闭场景细节、主角真实动机；不确定的信息要保留不确定感。

【字段规则】
- 帖子ID唯一且稳定。
- 发帖账号名写主角周围 NPC 的角色名、昵称或小号名；不要用普通路人、同城账号或无关公共号作为主贴发帖人，也不要承诺主页设定。
- 账号标签写账号气质，不写前端说明。
- 帖子标题可空；有标题也要短、生活化。
- 帖子正文像本人会发的动态，保留完整文本，不用分号压缩。
- 话题/附加信息可用英文分号分隔短项。
- 媒体信息只能填：无图、单图、三图、九宫格、截图流。
- 互动数据和时间要可信；不要每条都写得像全网爆了。

【留存】
广场保持约3-5条当前动态。达到5条后若需新增，默认先删除最老主贴，并同步删除两张评论表中同一帖子ID的评论；不要让广场固定成永久列表。

## sourceData.initNode

初始化时可插入3-5条当前公开动态，形成主角周围 NPC 的生活流氛围。内容应分散到不同 NPC 和不同生活面向；不要全部围绕主角，也不要写成论坛求助、新闻公告或剧情摘要。

## sourceData.deleteNode

主贴达到5条后新增时，优先从最老主贴开始删；删除主贴时同步删除精选评论表和普通评论分栏表中同一帖子ID的评论。

## sourceData.updateNode

一般情况下，不更新发过的帖子，等待攒满五条主贴就删除。

## sourceData.insertNode

当主角周围 NPC 出现新的公开生活片段、个人表达、照片感内容、世界侧影或轻社交互动时新增。每轮通常新增1-2条。优先补足“NPC 各自生活”的动态；只有发帖 NPC 合理知道且适合公开。

若新增前已有5条主贴，先按时间从最老主贴开始删除旧帖及其关联评论；新增后仍保持3-5条，不要让旧动态长期占满广场。

## sourceData.ddl

```sql
CREATE TABLE square_posts ( -- 广场主贴表
  row_id INTEGER PRIMARY KEY, -- 行号
  post_id TEXT NOT NULL UNIQUE, -- 帖子唯一标识
  author_name TEXT NOT NULL, -- 发帖账号名
  author_tag TEXT NOT NULL, -- 账号标签
  post_title TEXT, -- 帖子标题
  post_body TEXT NOT NULL, -- 帖子正文
  topic_meta TEXT, -- 话题/附加信息
  media_info TEXT NOT NULL CHECK(media_info IN ('无图', '单图', '三图', '九宫格', '截图流')), -- 媒体信息
  interaction_text TEXT NOT NULL CHECK(TRIM(interaction_text) <> ''), -- 互动数据
  time_text TEXT NOT NULL -- 时间文本
);
```
