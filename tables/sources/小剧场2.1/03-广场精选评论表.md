# 广场精选评论表

## sourceData.note

广场精选评论表记录动态流主贴下最值得展示、最有社交感的评论。一行代表一条精选评论，承担好友调侃、神回复、贴贴起哄、轻微吃瓜、当事人补刀或高赞共鸣的作用。它不是论坛楼层，不做长篇理性辩论。row_id 由框架自动维护，无需生成；下面的列编号从业务列开始计数。

【页面定位】
1. 精选评论应像朋友圈/动态下被顶上来的评论：短而有存在感，有关系感、有梗、有具体回应。
2. 评论必须围绕关联主贴的生活动态、照片内容、角色状态或当事人互动展开。
3. 可以玩梗、贴贴、轻微阴阳、求证、好友起哄，但不持续恶意攻击，不写成论坛辩论。
4. 评论者可为好友、同城路人、熟人小号、围观群众、当事人本人；账号名要像真实社交昵称。

【列定义】
- 列0: 关联帖子ID - 必须对应当前广场主贴表中已存在的帖子ID。
- 列1: 评论账号名 - 当前评论显示昵称，不额外承诺主页。
- 列2: 账号标签 - 表现评论者气质，如“好友起哄”“同城路过”“会接梗的人”“照片党”“当事人补刀”“贴贴党”。
- 列3: 评论正文 - 完整保留，不做分号压缩。应有明确回应点，不要只有“哈哈”“好甜”“支持”。
- 列4: 评论立场 - 只能填写支持、补充、吃瓜、玩梗、反对、求证。
- 列5: 点赞/回复数据 - 聚合文本，如“点赞892;回复34”。
- 列6: 时间文本 - 直接展示文本。
- 列7: 状态标签 - 如“高赞”“被楼主回应”“讨论较多”“封神评论”；其中金句、神回复可优先使用“封神评论”。


【数量与关联】
1. 每条主贴通常配 0 到 2 条精选评论，最多不超过 3 条。
2. 同一轮新增精选评论建议 1 到 3 条；优先给正在发酵或最有画面感的主贴补评论。
3. 如果关联主贴被删除，必须删除对应精选评论，禁止留下找不到主贴的孤儿评论。

## sourceData.initNode

初始化时可为主贴补 2 到 4 条精选评论，形成已有社交互动感。评论风格应覆盖好友起哄、路人吃瓜、补充照片细节、轻微求证等，不要全部写成论坛式长回复。

## sourceData.deleteNode

允许删除关联主贴已不存在、内容重复、已经过时、或存在恶意攻击倾向的精选评论。主贴删除时必须同步删除对应评论。

## sourceData.updateNode

当精选评论的点赞、回复、状态变化时更新。若评论被楼主回应、被围观点赞、变成神回复，可更新状态标签与互动数据；正文只在需要修正错别字、补充楼主回应或消除歧义时修改。

## sourceData.insertNode

当某条动态出现高存在感评论、好友起哄、精彩补充、贴贴发言、轻微吃瓜或明显被围观的回复时新增。每轮建议新增 1 到 3 条；新增前确认关联帖子ID存在，且同一主贴精选评论不要堆得过多。

## sourceData.ddl

```sql
CREATE TABLE square_featured_comments ( -- 广场精选评论表
  row_id INTEGER PRIMARY KEY, -- 行号
  post_ref TEXT NOT NULL, -- 关联帖子ID
  commenter_name TEXT NOT NULL, -- 评论账号名
  commenter_tag TEXT NOT NULL, -- 账号标签
  comment_body TEXT NOT NULL, -- 评论正文
  stance_tag TEXT NOT NULL CHECK(stance_tag IN ('支持', '补充', '吃瓜', '玩梗', '轻微反对', '求证')), -- 评论立场
  interaction_text TEXT NOT NULL CHECK(TRIM(interaction_text) <> ''), -- 点赞/回复数据
  time_text TEXT NOT NULL, -- 时间文本
  status_tag TEXT NOT NULL CHECK(status_tag IN ('高赞', '被楼主回应', '讨论较多', '封神评论')) -- 状态标签
);
```
